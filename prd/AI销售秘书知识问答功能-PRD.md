## 一、项目背景与目标

### 1.1 需求背景

- **当前痛点**：
  - **知识获取效率低**：销售人员在日常工作中遇到物流业务、产品服务、流程政策等问题时，需要手动翻阅文档或询问同事/主管，平均耗时约5分钟，严重影响销售跟进效率
  - **知识分散且不一致**：业务知识分散在产品手册、培训PPT、内部群聊等不同渠道，缺乏统一查询入口；不同销售对同一问题的理解可能存在偏差
  - **新人上手慢**：新入职销售需要记忆大量业务知识（如网络货运概念、易达宝使用方法、司机证件要求、平台费率政策等），培训周期长，人力成本高
  - **合规风险**：销售人员对流程政策、合规要求理解不一致，可能导致业务操作不规范，引发客户投诉或合规问题

- **为何引入 AI**：
  - 传统关键词搜索无法理解用户意图，搜索结果需要人工筛选和归纳
  - 规则引擎无法覆盖业务知识的海长尾场景和自然语言表达
  - LLM + RAG 能够理解自然语言问题，从知识库中精准检索并生成结构化答案，提供"对话式"的知识获取体验，大幅降低查询门槛

### 1.2 需求目标

- **量化目标**：
  - 知识查询效率：销售人员获取答案时间从平均 **5分钟缩短至30秒内**
  - 知识覆盖率：支持 **物流业务、产品服务、流程政策、合规风控、典型案例** 5大核心领域
  - 用户体验：支持流式输出，用户感知首字响应时间 **< 3秒**
  - 对话连续性：支持 **5轮上下文记忆**，支持多轮追问
  - 答案准确率：业务专家抽检通过率 **> 85%**
  - 幻觉率：**< 5%**

- **项目范围**：
  - **MVP 范围（Phase 1）**：
    - PC端 + APP端 AI销售秘书新增"知识问答"按钮入口
    - 基于现有RAG知识库接口的问答能力
    - 流式输出 + 5轮上下文记忆
    - 基础欢迎语 + 示例问题引导
  - **未来规划（Phase 2+）**：
    - 知识库自动更新机制
    - 答案来源引用展示
    - 语音输入/语音播报
    - 问答数据分析看板
    - 多模态问答（图片识别）

## 二、用户场景与交互流程

### 2.1 用户故事

| 角色 | 场景 | 期望结果 |
|------|------|----------|
| 一线销售（PC端） | 在办公室处理订单时，遇到客户询问"网络货运的税务政策是什么"，需要快速查询 | 点击知识问答按钮，输入问题，3秒内获得准确回答，可直接引用回复客户 |
| 一线销售（APP端） | 在外跑业务时，司机询问"跑网络货运需要哪些证件"，现场需要即时回答 | 打开APP端AI销售秘书，语音或文字输入问题，获得准确答复 |
| 新入职销售 | 刚入职1周，对业务不熟悉，需要学习"易达宝怎么用" | 通过知识问答功能自助学习，5轮内可追问细节，快速上手 |
| 销售主管 | 团队新成员频繁询问同类问题，希望有自助查询工具 | 团队成员通过知识问答自助解决，减少主管答疑时间 |

### 2.2 业务流程图

```mermaid
graph TD
    A[销售打开AI销售秘书] --> B[点击"知识问答"按钮]
    B --> C[系统显示欢迎语+示例问题]
    C --> D[销售输入问题]
    D --> E{问题是否合规?}
    E -->|否| F[返回拒答话术]
    E -->|是| G[调用RAG知识库检索]
    G --> H{检索到相关知识?}
    H -->|否| I[返回"未找到相关信息"兜底回复]
    H -->|是| J[LLM基于检索结果生成回答]
    J --> K[流式输出回答内容]
    K --> L{用户是否追问?}
    L -->|是,且≤5轮| M[携带上下文再次提问]
    M --> G
    L -->|否或>5轮| N[结束当前对话,清除上下文]
    F --> N
    I --> N
```

**AI 切入点**：Copilot 辅助驾驶模式 — 销售主动发起问答，AI 提供知识查询辅助，不自主发起对话或执行操作。

### 2.3 人在环 (HITL) 设计

- **触发条件**：
  - 用户对回答点踩（不满意）时，标记为 Bad Case
  - RAG 检索置信度低于阈值（< 0.6）时，提示用户"答案可能不够准确，请核实"
  - 用户问题超出知识库覆盖范围时，返回标准兜底话术

- **人工介入方式**：
  - 用户可编辑、复制 AI 回答内容
  - 用户可对回答进行"点赞/点踩"反馈
  - 点踩后可选择原因：答案不准确 / 答案不完整 / 答案不相关 / 其他

- **反馈闭环**：
  - **用户反馈收集**：点赞/点踩数据回流至数据服务，用于模型持续优化和知识库迭代
  - **Bad Case 沉淀**：用户点踩或标记的问答对自动归档，每周由业务专家审核，纳入黄金数据集用于模型调优
  - **知识库迭代**：Bad Case 中暴露的知识盲区，由知识运营人员补充至知识库

## 三、AI 核心任务定义

> **核心差异点**：传统 PRD 写功能逻辑 (If-Then)，AI PRD 写任务定义 (Input-Output)。

### 3.1 任务描述

- **任务类型**：RAG 问答（检索增强生成）

- **输入字段定义**：

  | 字段 | 类型 | 说明 |
  |------|------|------|
  | user_question | string | 用户输入的问题文本，长度 1-500 字符 |
  | conversation_history | array | 当前对话历史，最多5轮，每轮包含 role（user/assistant）和 content（文本内容） |
  | session_id | string | 会话唯一标识，用于上下文关联 |
  | user_id | string | 用户唯一标识 |
  | platform | string | 终端类型：PC / APP |
  | timestamp | string | 请求时间戳，格式 `YYYY-MM-DD HH:mm:ss` |

- **输出字段定义**：

  | 字段 | 类型 | 说明 |
  |------|------|------|
  | answer | string | AI 生成的回答文本，支持 Markdown 格式，长度通常 50-1000 字符 |
  | is_fallback | boolean | 是否为兜底回复（检索结果为空时返回固定话术） |
  | confidence | number | 回答置信度，0-1 之间，< 0.6 时前端提示用户核实 |
  | session_id | string | 会话唯一标识，与请求一致 |
  | turn_count | number | 当前对话轮次，1-5，超过5轮后重置 |

- **输入 (Input)**：用户的自然语言问题 + 最多5轮的对话历史（每轮包含用户问题和AI回答）
- **输出 (Output)**：基于 RAG 知识库检索结果生成的回答文本（支持 Markdown 格式），以流式方式逐字/逐段返回

### 3.2 模型选型与配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 基座模型 | Qwen-Plus / Claude Sonnet | 兼顾中文理解能力、推理能力和成本效益；若已有RAG接口使用特定模型，则沿用 |
| 温度 (Temperature) | 0.3 | 知识问答需要准确性，降低随机性 |
| Top P | 0.9 | 控制输出多样性 |
| 最大输出长度 | 2000 tokens | 覆盖大部分问答场景，超长回答需截断并提示"回答过长，已截断" |
| 上下文窗口 | 16k tokens | 支持5轮对话历史 + 检索文档 + 系统提示词 |

### 3.3 提示词策略

- **System Prompt（角色设定）**：

```
你是"AI销售秘书"的知识问答助手，专门为物流行业销售人员提供业务知识查询服务。

你的能力范围包括：
1. 物流业务知识（如网络货运、仓储管理、配送流程等）
2. 产品服务介绍（如易达宝、平台功能等）
3. 流程政策说明（如司机证件要求、准入标准等）
4. 合规风控指南（如税务政策、安全规范等）
5. 典型案例参考

回答要求：
- 仅基于提供的知识库内容回答，不要编造信息
- 如果知识库中没有相关信息，请回复："抱歉，我暂时没有找到相关信息，请稍后重试或咨询主管。"
- 回答应简洁明了，重点突出，适合销售人员快速理解
- 使用 Markdown 格式组织内容（标题、列表、加粗等）
- 如问题涉及操作类指引，请使用步骤式说明
- 语气专业、友好，避免使用过于技术化的术语

当前对话轮次：{turn_count}/5
```

- **Few-Shot Examples（少样本示例）**：

  | 场景 | 输入 | 输出 |
  |------|------|------|
  | 产品咨询 | "网络货运是什么？" | "网络货运是指...\n\n**核心特点：**\n- 特点1...\n- 特点2...\n\n**适用场景：**..." |
  | 操作指引 | "易达宝怎么用？" | "使用易达宝的步骤如下：\n1. 第一步...\n2. 第二步...\n\n**注意事项：**..." |
  | 政策查询 | "司机需要哪些证件？" | "从事网络货运的司机需要准备以下证件：\n1. ...\n2. ...\n\n**办理建议：**..." |
  | 超出范围 | "今天天气怎么样？" | "抱歉，我主要为您提供物流业务相关的知识问答服务，暂时无法回答天气相关问题。您可以问我关于网络货运、易达宝、司机证件、费率政策等方面的问题。" |

- **CoT（思维链）要求**：

```
请按以下步骤生成回答：
1. 理解用户问题意图
2. 基于检索到的知识库内容，提取相关信息
3. 组织回答结构（定义/概念 → 详细说明 → 注意事项/建议）
4. 检查回答是否仅基于知识库，无编造内容
5. 输出最终回答
```

### 3.4 知识库与 RAG 策略（仅针对知识密集型任务）

- **数据源**：现有 RAG 知识库接口（已可用），覆盖物流业务、产品服务、流程政策、合规风控、典型案例等领域

- **切片策略 (Chunking)**：
  - 按语义/段落切分，每段 500-800 tokens
  - 段落间重叠 10-15%，避免关键信息被截断
  - 支持 Markdown/HTML 格式文档的结构化解析

- **检索策略**：
  - 混合检索：关键词检索（BM25）+ 向量检索（Embedding）
  - 检索 Top 5 相关文档片段
  - 支持重排序（Rerank），优先返回高置信度片段
  - 检索结果置信度阈值：≥ 0.6

- **上下文管理策略**：
  - 后端维护对话上下文，最多保留 5 轮历史
  - 每次请求携带完整对话历史 + 当前问题
  - 超过 5 轮后，自动清除历史，从第 1 轮重新开始
  - 用户可手动点击"新对话"按钮清除上下文

## 四、功能需求

### 4.1 输入端

- **问题输入**：
  - 支持文本输入，长度 1-500 字符
  - 支持发送按钮或回车键发送（APP 端支持语音输入，Phase 2）
  - 输入框占位符："请输入您的问题，如：网络货运是什么？"

- **预处理**：
  - 输入文本去除首尾空格
  - 空输入禁止发送，提示"请输入问题"
  - 敏感关键词过滤（政治、色情、暴力等），触发后返回标准拒答话术

### 4.2 输出端与交互

- **欢迎语展示**：
  - 点击"知识问答"按钮后，显示固定欢迎语：
    > 您好，我可以帮您查询和解答物流业务知识、产品服务、流程政策、合规风控及典型案例等问题。
    >
    > 您可以直接问我：网络货运是什么、易达宝怎么用、司机需要哪些证件、平台费率政策是什么等等。

- **流式输出 (Streaming)**：
  - 采用 SSE（Server-Sent Events）协议实现流式输出
  - 前端实现打字机效果，逐字/逐词展示回答内容
  - 首字延迟（TTFT）目标：< 3 秒
  - 流式输出过程中显示加载动画（如光标闪烁）

- **对话历史展示**：
  - 以对话气泡形式展示用户问题和 AI 回答
  - 用户问题靠右，AI 回答靠左
  - 支持 Markdown 格式渲染（标题、列表、加粗、代码块等）

- **上下文记忆**：
  - 保持 5 轮对话上下文
  - 每轮对话显示轮次标识（如"第 1/5 轮"）
  - 第 5 轮结束后，提示"当前对话已达5轮上限，开启新对话以继续提问"
  - 提供"新对话"按钮，手动清除上下文

- **加载状态**：
  - RAG 检索和 LLM 生成过程中，显示"思考中..."或加载动画
  - 超时（> 15秒）时显示"回答生成超时，请重试"

- **示例问题推荐**（可选）：
  - 欢迎语下方展示可点击的示例问题标签：
    - "网络货运是什么？"
    - "易达宝怎么用？"
    - "司机需要哪些证件？"
    - "平台费率政策是什么？"

### 4.3 异常处理与兜底

- **拒答机制**：

  | 场景 | 回复话术 |
  |------|----------|
  | 超出业务范围 | "抱歉，我主要为您提供物流业务相关的知识问答服务。您可以问我关于网络货运、易达宝、司机证件、费率政策等方面的问题。" |
  | 敏感/违规内容 | "抱歉，我无法回答该问题。" |
  | 空输入 | "请输入您的问题。" |

- **幻觉抑制**：
  - RAG 检索结果为空或置信度 < 0.6 时，**严禁 AI 编造答案**
  - 标准兜底回复："抱歉，我暂时没有找到相关信息，请稍后重试或咨询主管。"
  - 兜底回复时不显示置信度提示

- **超时处理**：
  - 请求超过 15 秒未返回时，显示"回答生成超时，请稍后重试"
  - 同时记录超时日志，用于后续性能优化

- **网络异常**：
  - 网络断开时，显示"网络连接异常，请检查网络后重试"
  - 支持重试按钮

## 五、评测与验收标准

> 这是 AI 项目能否上线的生死线，必须量化。

### 5.1 黄金数据集

- **构建方式**：由物流业务专家人工编写的 **80 条**标准问答对 (Ground Truth)
- **覆盖范围**：
  - **70% 常见场景（56条）**：
    - 物流业务知识（14条）：网络货运定义、优势、模式等
    - 产品服务介绍（14条）：易达宝、平台功能等
    - 流程政策说明（14条）：司机证件、准入标准、操作流程等
    - 合规风控指南（7条）：税务政策、安全规范等
    - 典型案例参考（7条）：典型客户案例、场景案例等
  - **30% 极端场景（24条）**：
    - 多轮追问场景（8条）：基于上一轮答案继续提问
    - 模糊/口语化问题（8条）：如"那个运费怎么算来着？"
    - 超出范围问题（4条）：如"今天天气怎么样？"
    - 敏感/违规问题（4条）：测试拒答机制

### 5.2 验收指标

- **自动化评测 (LLM-as-a-Judge)**：
  - 准确性：**> 90%**（回答与标准答案核心信息一致）
  - 相关性：**> 95%**（回答与用户问题高度相关）
  - 完整性：**> 85%**（回答覆盖标准答案的关键信息点）

- **人工评测**：
  - 可用性：业务专家抽检通过率 **> 85%**
  - 幻觉率：**< 5%**（回答中包含知识库中不存在的事实）
  - 拒答准确率：**> 95%**（对超出范围/敏感问题的拒答准确率）

- **性能指标**：
  - 首字延迟 (TTFT)：**< 3 秒**（P95）
  - 全量生成时间：**< 15 秒**（单轮问答，P95）
  - 服务可用性：**> 99.9%**
  - 并发支持：≥ 50 QPS

### 5.3 上线后监控

- **日常指标**：
  - 每日调用量、活跃用户数
  - 失败率（超时、异常）趋势，告警阈值 > 5%
  - 用户满意度（点赞/点踩比），目标 > 4:1
  - 平均对话轮次（目标 2-3 轮/会话）
  - 幻觉率监控（通过 Bad Case 抽检），告警阈值 > 5%

- **Bad Case 巡检**：
  - 频率：**每周一次**
  - 负责人：产品运营 + 业务专家
  - 从发现到修复的 SLA：**高优 Bad Case 3 个工作日内修复**

- **告警机制**：
  - 当失败率 > 5% 或幻觉率 > 5% 时，自动发送告警通知（钉钉/企微）
  - 连续 3 次告警未恢复，触发人工介入，临时下线知识问答功能

## 六、数据安全与合规

- **数据隐私**：
  - 用户 ID、会话 ID 等标识信息需脱敏存储，日志中不保留明文
  - 问答数据仅用于模型优化和知识库迭代，不用于其他商业用途
  - 若涉及个人信息，需符合《个人信息保护法》要求

- **内容安全**：
  - **输入过滤**：用户问题需经过敏感词过滤（政治、色情、暴力、竞品等关键词），触发后返回标准拒答话术
  - **输出过滤**：AI 生成回答需经过安全审核层，确保不输出违规内容
  - 过滤规则库由法务/合规团队维护，每月更新

- **知识库安全**：
  - 知识库内容需经过审核后方可上线，确保信息准确性和合规性
  - 内部机密信息（如未公开的费率、商业策略等）不得录入知识库
  - 知识库访问权限控制：仅授权人员可编辑，AI 仅可读

- **算法合规**：
  - 符合《生成式人工智能服务管理暂行办法》要求
  - 在知识问答界面明确告知用户"AI 回答仅供参考，不构成最终业务指导"
  - 提供用户反馈入口（点赞/点踩），便于持续优化
