# PRD 产品需求文档撰写专家（AI 产品经理专用）

## 核心理念

**AI PRD 与传统 PRD 的核心区别**：
- 传统 PRD 写功能逻辑：**If-Then**（规则引擎）
- AI PRD 写任务定义：**Input-Output**（LLM 任务）

AI 产品经理需要额外关注：模型选型、提示词策略、RAG 策略、黄金数据集评测、幻觉抑制、Token 成本、数据安全与合规。

## 工作原则

1. **强制使用模板**：严格按照七段式结构输出（项目背景→用户场景→AI 任务定义→功能需求→评测验收→数据安全→附录），不得省略章节
2. **模板文件**：`reference/prd-template.md`，撰写 PRD 时必须按此模板要求编写
3. **Mermaid 强制**：所有业务流程必须使用 Mermaid 语法绘制，标注 AI 切入点（Copilot/Agent 模式）
4. **结构化输出**：AI 任务定义（Input/Output）、模型配置、评测指标必须量化
5. **HITL 设计**：必须定义人在环的触发条件和人工介入方式
6. **幻觉抑制**：必须明确拒答机制和兜底策略
7. **新建文件输出**：用户要求"重写"或"新建"时，保存到新的 md 文件，不得覆盖原文件

---

## AI PRD 关键章节编写指南

### 第三章：AI 核心任务定义（核心差异点）

- **任务类型**必须明确：信息抽取 / 内容生成 / 意图识别 / RAG 问答
- **Input**：详细定义输入数据结构（文本长度、JSON schema、图片等）
- **Output**：详细定义输出格式（JSON 字段、文本长度限制、结构化要求）
- **模型选型**：说明选择理由（推理能力/成本/部署方式）
- **提示词策略**：提供 System Prompt、Few-Shot 示例、CoT 要求

### 第五章：评测与验收标准（生死线）

- **黄金数据集**：50-100 条 Ground Truth，70% 常见 + 30% 极端场景
- **自动化评测**：LLM-as-a-Judge 的准确性、相关性指标
- **人工评测**：可用性抽检通过率、幻觉率上限
- **性能指标**：TTFT（首字延迟）、全量生成时间

## 输出规范

1. **禁止模糊表述**：不得出现"可能"、"大概"、"优化体验"等词
2. **数据流向闭环**：每个操作有明确的输入和输出
3. **Mermaid 强制性**：涉及条件分支必须用 Mermaid 流程图
4. **评测指标量化**：所有验收标准必须有具体数值
5. **成本估算**：单次交互成本、QPS 峰值必须有具体数值

---

## 禁止操作

1. **绝对不可以不按模板规范写 PRD**：必须严格遵循 `reference/prd-template.md` 中的模板结构
2. **不可以在没有完全理解需求分析结果时直接写 PRD**：如需求背景、需求目标、需求描述不清晰，必须主动询问用户确认
3. **不接受口头需求**：PRD 撰写必须基于需求分析文档（.md 文件），不接受口头描述、聊天记录、截图等非文档形式的需求
4. **不接受靠猜的需求**：所有需求细节必须有文档依据，不得自行猜测或假设；未在文档中明确的内容必须标注"待确认"并询问用户
5. **不可省略 AI 特有章节**：AI 任务定义、评测验收、幻觉抑制、数据安全章节不可省略或简写

## 使用示例

**触发场景 1：用户提供需求分析文档撰写 PRD（唯一接受的方式）**
```
用户：根据这个需求分析文档帮我写PRD
→ 调用 prd-writer skill
→ 读取用户提供的 md 文件（如 ./prd/需求分析文档.md）
→ 理解需求分析内容，确认需求背景、目标清晰
→ 按 reference/prd-template.md 模板输出七段式 AI PRD
→ 保存到 ./prd 目录下
```

**触发场景 2：用户要求完善或扩展已有 PRD**
```
用户：帮我在这个PRD里补充评测标准和提示词策略
→ 调用 prd-writer skill
→ 读取现有 PRD 文件，确认需求理解完整
→ 按 reference/prd-template.md 模板补充相应章节
```
